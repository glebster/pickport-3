# 09 — Узкие места и открытые вопросы

## Узкие места

1. **Качество классификации ИИ.**
   - Риск: неверный `action + object`, пользователи теряют доверие.
   - Митигирование: намерения, fallback `noted + note`, ручной выбор рецепта.

2. **Перегрузка интерфейса.**
   - Риск: слишком много полей/категорий.
   - Митигирование: прогрессивное раскрытие, минимум обязательных.

3. **Оффлайн‑синхронизация.**
   - Риск: конфликтующие правки и дубли.
   - Митигирование: лог изменений, детект дублей, стратегия resolve.

4. **Приватность данных.**
   - Риск: данные личные и чувствительные.
   - Митигирование: шифрование, экспорт/удаление, прозрачные настройки.

5. **Стоимость AI‑обработки.**
   - Риск: высокие расходы при распознавании фото.
   - Митигирование: кеширование, батчи, настройка лимитов.

6. **Слишком широкая семантика.**
   - Риск: трудно объяснить пользователю смысл «Action + Object».
   - Митигирование: понятные примеры, скрытая семантика в UI.

7. **Совместные группы и приватность.**
   - Риск: пользователь путается, какие записи общие, какие приватные.
   - Митигирование: явные метки «личное/общее», отдельные разделы и лог доступа.

## Открытые вопросы

- Нужен ли отдельный объект для эмоций или достаточно тегов?
- Какой минимальный набор категорий стартует без перегрузки?
- Нужно ли позволять пользователям самим создавать новые `action`/`object`?
- Какой уровень доверия к AI достаточен, чтобы не раздражать?
- Нужен ли sharing отдельной карточки без раскрытия всей истории?

## Следующие решения для команды

1. Утвердить список MVP‑категорий.
2. Зафиксировать структуру рецептов и обязательные поля.
3. Определить подход к оффлайн‑синхронизации.
4. Определить минимальный набор AI‑функций для запуска.
